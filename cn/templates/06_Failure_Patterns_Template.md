## 0. 预部署检查清单

<!-- 说明：最关键的章节。包含可在部署开始前验证环境的可执行 bash 脚本。
     每项检查必须：
     1. 有清晰的标题
     2. 包含完整的、可直接复制粘贴的 bash 脚本
     3. 展示预期的成功输出
     4. 包含失败时的修复步骤

     覆盖范围：连通性、网络/NACL/安全组、镜像、密钥、负载均衡器。
     按从基础（连通性）到具体（应用）的顺序排列。 -->

**⚠️ 关键**：在每个环境部署之前运行此检查清单，以避免常见故障。

### 0.1 基础设施验证

<!-- 说明：设置所有后续检查使用的环境变量。 -->

```bash
# 设置环境变量
ENV="[environment]"  # 修改为：dev、test、staging、prod
REGION="[region]"

echo "=== $ENV 环境的预部署检查清单 ==="

# 验证 AWS 凭证和区域
aws sts get-caller-identity --region $REGION --query '{Account:Account,Region:Region}'
```

**✅ 预期结果**：AWS 账户 ID 和区域已确认

### 0.2 连通性检查

<!-- 说明：验证部署工具（SSM、SSH、kubectl）能否连接到目标实例/节点。 -->

```bash
# 获取该环境的所有实例
INSTANCES=$([command to list instances for the environment])

echo "发现的实例：$INSTANCES"

# 检查每个实例的连通性
for instance in $INSTANCES; do
  STATUS=$([command to check instance connectivity])
  echo "实例 $instance 状态：${STATUS:-NOT_CONNECTED}"
done
```

**✅ 预期结果**：所有实例应显示 `[Online/Ready/Connected]`

### 0.3 网络和安全组验证

<!-- 说明：最常见的故障类别。验证端口、NACL 规则和路由。
     检查所有涉及的安全组，而不仅仅是主安全组。
     常见错误是忽略辅助组件（ETCD、监控）使用不同安全组的情况。 -->

```bash
# 验证安全组规则中所需端口是否开放
SG_ID=$([command to find security group])
echo "安全组：$SG_ID"

# 检查所需端口
REQUIRED_PORTS=([port1] [port2] [port3])
for port in "${REQUIRED_PORTS[@]}"; do
  RULE=$([command to check if port is open])
  if [ "$RULE" == "[]" ]; then
    echo "  ❌ 端口 $port：未配置"
  else
    echo "  ✅ 端口 $port：已配置"
  fi
done
```

**✅ 预期结果**：所有所需端口应显示 `已配置`

<!-- 说明：如果组件分布在多个安全组中，需为每个安全组分别包含独立的子检查。 -->

### 0.4 密钥和配置检查

<!-- 说明：验证所有密钥/参数是否存在，防止因缺少密钥导致启动失败
     而在部署中途崩溃。 -->

```bash
# 验证密钥/参数是否存在
SECRET_NAME="[secret-name-pattern-${ENV}]"
[command to check secret existence] || echo "❌ 密钥 $SECRET_NAME 未找到"
```

**✅ 预期结果**：所有密钥/参数应存在且具有有效的 ARN

### 0.5 镜像/制品可用性检查

<!-- 说明：验证目标环境可以访问 Docker 镜像或制品。
     参考已验证可用镜像表。 -->

```bash
# 验证所需镜像/制品是否可用
IMAGES=(
  "[image1:tag]"
  "[image2:tag]"
  "[image3:tag]"
)

for image in "${IMAGES[@]}"; do
  echo "检查 $image..."
  [command to verify image availability] && echo "  ✅ 可用" || echo "  ❌ 不可用"
done
```

**✅ 预期结果**：所有镜像/制品应显示 `可用`

### 0.6 负载均衡器/目标组检查

<!-- 说明：验证目标组是否存在。目标可能尚未注册（这发生在部署过程中），
     但目标组本身应该存在。 -->

```bash
# 检查目标组和已注册的目标
TG_ARN=$([command to find target group])

if [ -n "$TG_ARN" ]; then
  echo "目标组：$TG_ARN"
  TARGETS=$([command to list registered targets])
  if [ -z "$TARGETS" ]; then
    echo "  ⚠️ 尚未注册目标（部署前属于正常情况）"
  else
    echo "  ✅ 现有目标：$TARGETS"
  fi
else
  echo "❌ 未找到 $ENV 的目标组"
fi
```

**✅ 预期结果**：目标组存在（部署前目标可能为空）

### 问题 2.1：NACL 规则顺序导致环境隔离失效

<!-- 说明：完整的故障模式示例，展示了期望的质量水平。 -->

**症状**：
```
# 预期：连接到另一个环境应该失败
$ timeout 3 bash -c "echo > /dev/tcp/[other-env-ip]/[port]"
# 实际：连接成功（隔离已失效！）
```

**根本原因**：
NACL 规则按规则编号顺序评估，第一条匹配的规则生效。如果一条允许所有流量（0.0.0.0/0）的规则排在拒绝其他环境的规则之前，那么针对跨环境流量的拒绝规则将永远不会被评估。

**有问题的配置**：
<!-- 说明：将错误配置与修复方案并列展示，让工程师可以
     对比当前状态并立即识别问题。 -->
```
规则 50：  allow 0.0.0.0/0     ← 最先匹配，允许所有流量
规则 200： deny 10.0.20.0/24   ← 对 10.0.20.x 流量永远不会被评估
```

**修正后的配置**：
```
规则 100： allow [own-subnet-CIDR]   （本环境）
规则 200： deny [other-env-CIDR-1]   （在允许所有之前被评估）
规则 201： deny [other-env-CIDR-2]
规则 500： allow 0.0.0.0/0           （NAT GW 返回流量，在拒绝之后被评估）
规则 32767： deny 0.0.0.0/0          （默认拒绝）
```

**解决方案**：
```bash
# 删除编号较低的允许规则
aws ec2 delete-network-acl-entry \
  --network-acl-id [NACL_ID] \
  --rule-number 50 \
  --ingress

# 在拒绝规则之后添加（例如，规则 500）
aws ec2 create-network-acl-entry \
  --network-acl-id [NACL_ID] \
  --rule-number 500 \
  --protocol -1 \
  --rule-action allow \
  --ingress \
  --cidr-block 0.0.0.0/0
```

**预防措施**：
- 始终使用大于 400 的规则编号作为 allow 0.0.0.0/0 规则
- 每次修改 NACL 后测试环境隔离性
- 将隔离测试纳入部署验证检查清单

## 部署顺序与检查点

<!-- 说明：推荐带有明确检查点的部署顺序。
     许多故障发生在阶段执行顺序不正确时。每个阶段需要：
     1) 清晰的名称，2) 具体的操作，3) 带有验证命令的检查点，
     4) 预期结果。

     检查点是一个停止点。在通过之前不要继续下一步。
     这可以防止故障在阶段之间级联传播。

     各平台的阶段结构：
     EC2/VM：预验证 → 基础服务 → 主应用 → 次要服务 → 负载均衡器
     容器：预验证 → 基础设施 → IAM → 容器 → 负载均衡器 -->

### 推荐部署顺序

```
阶段 1：预部署验证（第 0 节）
├── ✅ 0.1 基础设施验证
├── ✅ 0.2 连通性检查
├── ✅ 0.3 网络和安全组验证
├── ✅ 0.4 密钥和配置检查
└── ✅ 0.5 镜像/制品可用性检查

阶段 2：[组件 1] 部署
├── 部署 [组件名称]
├── ⏸️ 检查点：验证 [组件] 健康状态
│   └── [验证命令]
└── 预期：[成功的表现]

阶段 3：[组件 2] 部署
├── 部署 [组件名称]
├── ⏸️ 检查点：验证 [组件] 健康状态
│   └── [验证命令]
└── 预期：[成功的表现]

阶段 4：[组件 N] 部署
├── 部署 [组件名称]
├── 将目标注册到负载均衡器
├── ⏸️ 检查点：验证负载均衡器健康状态
│   └── [验证命令]
└── 预期：所有目标显示健康

阶段 5：部署后验证
├── 通过 [负载均衡器/服务端点] 测试连接
├── 验证日志已流式传输到 [CloudWatch/日志系统]
└── 在 [跟踪系统] 中记录部署结果
```

<!-- 说明：ASCII 树形字符：├── 分支、└── 最后一项、⏸️ 检查点、✅ 必须通过。
