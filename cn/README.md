## 1. Spec Coding 规范编码方法论概述

### 什么是 Spec Coding？

Spec Coding（规范编码）是一种**以结构化规范驱动编码过程**的开发实践。工程师在动手实现之前，先通过一系列相互关联的文档定义问题、需求、设计和任务分解。

**核心原则**: 先想清楚再编码，先写规范再动手。

### 为什么需要 Spec Coding？

| 收益 | 说明 |
|------|------|
| **减少返工** | 在纸面上发现设计错误的成本是代码阶段的十分之一 |
| **提升对齐** | 跨团队评审规范可以避免沟通偏差 |
| **审计追溯** | 每个决策都有记录和理由 |
| **驱动自动化** | 结构化规范可以驱动自动化任务执行 |
| **加速上手** | 新工程师通过规范理解系统，无需代码考古 |

### 适用场景

- 基础设施和平台工程项目
- 需要跨团队协调的功能开发
- 有合规或审计要求的系统
- 失败成本较高的项目
- 跨多个迭代周期（Sprint）的项目

## 2. 四文档流水线

Spec Coding 遵循顺序流水线，每个文档基于前一个文档构建：

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│                 │     │                 │     │                 │     │                 │
│  1. GAP 差距分析 │────>│ 2. 需求规格说明 │────>│  3. FIP 功能    │────>│  4. 任务清单    │
│                 │     │                 │     │  实现计划       │     │                 │
│ "现状是什么？"  │     │ "要建什么？"    │     │ "怎么设计？"    │     │ "怎么执行？"    │
│                 │     │                 │     │                 │     │                 │
│ "差距在哪里？"  │     │ "怎么验收？"    │     │                 │     │ "按什么顺序？"  │
│                 │     │                 │     │                 │     │                 │
└─────────────────┘     └─────────────────┘     └─────────────────┘     └─────────────────┘
        |                       |                       |                       |
        v                       v                       v                       v
  评估现状，明确          定义验收标准，          含 Mermaid 图的         按阶段划分任务，
  差距与投资方向          非功能需求，            架构设计与风险评估      含依赖与工作量估算
                          基础设施需求
```

### 流水线流程

| 阶段 | 文档 | 输入 | 输出 | 评审人 |
|------|------|------|------|--------|
| 1 | GAP 差距分析 | 业务问题、现有基础设施 | 差距清单、投资论证 | 技术负责人、产品负责人 |
| 2 | 需求规格说明 | GAP 分析结论 | 功能/非功能/基础设施/集成需求 | 技术负责人、QA、安全 |
| 3 | FIP 功能实现计划 | 需求规格说明 | 架构设计、实现方案、风险评估 | 架构师、高级工程师 |
| 4 | 任务清单 | FIP 实现方案 | 分阶段任务（含依赖关系、工作量估算） | 工程团队 |

## 5. 快速开始

### 第一步：将模板复制到你的项目

```bash
# 将整个 templates 目录复制到你的项目中
cp -r spec-coding-templates/templates/ /path/to/your/project/docs/specs/

# 或按需复制单个模板
cp spec-coding-templates/templates/01_GAP_Analysis_Template.md \
   /path/to/your/project/docs/GAP_MyProject.md
```

### 第二步：遵循流水线

```
1. 开始 ────→ 填写 GAP 差距分析（模板 01）
   │              评估现状，识别差距
   │
2. 下一步 ──→ 编写需求规格说明（模板 02）
   │              将差距转化为可度量的需求
   │
3. 然后 ────→ 创建 FIP 功能实现计划（模板 03）
   │              设计架构，规划实现方案
   │
4. 最后 ────→ 编写任务清单（模板 04）
   │              分解为分阶段的、可跟踪的任务
   │
5. 全程贯穿→ 应用命名规则（模板 05）
              参考故障模式（模板 06）
              配置 AUTO_TASK_CONFIG（模板 07）
              执行基础设施依赖规则（模板 08）
```

### 第三步：填写与评审

每个模板遵循相同的模式：
1. 将 `[PLACEHOLDER]` 占位符替换为项目实际内容
2. 阅读并遵循 `<!-- INSTRUCTION: ... -->` 指令注释，完成后删除
3. 使用一致的风险等级标识：`🔴 严重`、`🟡 高`、`🟢 中`
4. 用以下标记跟踪状态：`✅`（完成）、`❌`（待处理）、`🔄`（进行中）、`⏳`（受阻）

## 6. 模板使用指南

### 占位符标记

所有模板使用 `[PLACEHOLDER]` 占位符表示需要替换的项目内容：

```markdown
<!-- 填写前 -->
**Issue**: #[ISSUE_NUMBER] - [Issue Title]
**Branch**: [BRANCH_NAME]
**Author**: [AUTHOR_NAME]

<!-- 填写后 -->
**Issue**: #1880 - ALB + Fargate Infrastructure for Java Container Deployment
**Branch**: feature/1880-alb-fargate-java-card-service
**Author**: arthurren
```

### 指令注释

模板中嵌入了 HTML 注释形式的指导说明。阅读并遵循后删除：

```markdown
<!-- INSTRUCTION: 提供 3-5 个要点描述差距。
     每个要点应说明缺少什么以及为什么重要。
     填写完成后删除此注释。 -->
```

### 风险等级体系

所有模板使用一致的风险等级标识：

| 等级 | 含义 | 需要的操作 |
|------|------|------------|
| 🔴 严重 | 阻碍交付 | 必须在继续之前解决 |
| 🟡 高 | 必需但不立即阻碍 | 计划解决 |
| 🟢 中 | 重要但可延期 | 迭代处理 |

### 状态标记

| 标记 | 含义 |
|------|------|
| ✅ | 已完成 / 已通过 |
| ❌ | 待处理 / 未完成 |
| 🔄 | 进行中 |
| ⏳ | 受阻 / 等待中 |
| 🚫 | 已取消 / 已移除 |

### 已填写模板的文件命名规范

```
{文档类型}_{项目名称}.md

示例:
GAP_ALB_Fargate_java_service.md
REQ_1880_alb_fargate_java_service.md
FIP_1928_pipeline_improvement.md
TASK_LIST_1880_alb_fargate_java_service.md
NAMING_card_service_v2.md
```

**背景**: 在 ECS Fargate 上部署新的 Java 微服务，包含 ALB、API Gateway 和 VPC Link。

      │                                       │
      │                                       ├── 命名规则（用于所有资源）
      │                                       ├── 故障模式（风险评估）
      │                                       └── 基础设施依赖规则（组件排序）
      │
      └── 记录现有基础设施差距：
          "没有容器编排平台"
          "没有标准化的 Java 服务部署流水线"
```

**使用模板**: 01 + 02 + 03 + 04 + 05 + 06 + 07 + 08（全部模板）

**背景**: 修复开发环境 PostgreSQL 集群的 NLB 路由问题。

```

**使用模板**: 02（部分）+ 03（部分）+ 04（部分）

**背景**: 在所有微服务之间实施 mTLS。

```

      └── 评估现有安全态势                      └── 参考故障模式

```

      │
      └── 定义所有资源类型的命名模式
          创建验证脚本（第 6 节）
          规划现有资源的迁移（第 7 节）

### 场景 E：多环境部署自动化

**背景**: 自动化 Fargate 服务在 dev、staging、prod 之间的部署。

```

      │                    │
      └── 定义环境提升      └── 配置带验证关卡的
          顺序和阻断条件        每环境自动化执行
```

**使用模板**: 08 + 07

## 10. 最佳实践

### 规范编写

1. **从 GAP 分析开始**，即使是"显而易见"的项目 — 这一步往往能发现隐藏的差距
2. **至少 2 名团队成员评审**模板后再进入实现阶段
3. **保留指令注释**直到模板全部填写完成，然后再删除
4. **在一个项目中使用一致的风险等级**标识
5. **将每份规范关联到 GitHub/Jira Issue**，确保可追溯

### 文档生命周期

| 阶段 | 状态 | 含义 |
|------|------|------|
| 编写 | `Draft` | 初始创建，正在编写 |
| 评审 | `Under Review` | 同行评审阶段 |
| 批准 | `Approved` | 设计已获得签字确认 |
| 实现 | `Implementation In Progress` | 工程团队正在执行 |
| 完成 | `Complete` | 已交付并验证通过 |

### 版本控制

- 将已填写的规范与代码变更一起提交
- 实现偏离计划时更新规范
- 利用每份 FIP 末尾的 `Review History` 表格跟踪变更
- 每次重大更新时递增文档版本号

### 常见反模式

| 反模式 | 为什么有问题 | 正确做法 |
|--------|-------------|----------|
| 跳过 GAP 分析 | 遗漏隐藏需求 | 始终先评估现状 |
| 不读指令就复制模板 | 产出空泛、未填写的规范 | 逐条阅读每条 INSTRUCTION 注释 |
| 先写代码再补规范 | 文档形式主义 | 先写规范再编码 |
| 变更后从不更新规范 | 规范变得误导 | 实现偏离时及时更新 |
| 一个人写规范 | 盲点、偏见 | 至少 2 人评审 |
| 小任务也走全套流程 | 浪费时间在仪式上 | 使用精简流水线（场景 B） |

## 11. 模板详细参考

### 模板 01：GAP 差距分析（约 1,073 行）
- **章节**: 概述、现状评估、差距分析矩阵、战略路线图、风险评估、成功指标
- **关键交付物**: 含量化差距的投资论证报告

### 模板 02：需求规格说明（约 1,022 行）
- **章节**: 摘要、功能需求（FR-X.X.X）、非功能需求、基础设施需求、集成需求、部署需求、测试需求、验收标准
- **关键交付物**: 含验收标准勾选框的可度量需求

### 模板 03：功能实现计划 FIP（约 990 行）
- **章节**: 架构设计（Mermaid）、详细设计、安全设计、性能设计、风险评估、实现方案、测试策略、监控、发布计划
- **关键交付物**: 可直接进入实现阶段的完整技术设计

### 模板 04：任务清单（约 843 行）
- **章节**: 阶段划分、含依赖的任务表、工作量估算、关键路径、每阶段验收标准
- **关键交付物**: 责任明确、顺序清晰的可行任务列表

### 模板 05：命名规则（约 685 行）
- **章节**: 通用原则、资源命名规则（10 种资源类型）、环境后缀规则、标签标准、验证脚本、迁移指南
- **关键交付物**: 可通过 CI/CD 验证执行的命名标准

### 模板 06：故障模式（约 650 行）
- **章节**: 故障分类、模式库、根因分析模板、恢复流程、预防检查清单
- **关键交付物**: 已知故障模式的运维手册

### 模板 07：自动任务配置（约 550 行）
- **章节**: 配置模板（YAML）、字段参考、使用示例、与任务清单集成、自定义指南、故障排除
